## 2. Rejection risks (will likely bounce in review)

1. **Trademarked league names in keywords** — `nfl,nba,mlb,nhl` (and arguably "college"). Guideline 2.3.7; using marks you don't own is a common metadata rejection. **Remove them** (new keyword set in §4). _High._
2. **Review-gating funnel** (`ReviewPromptSheet.swift`) — the "Enjoying The Sideline?" sheet routes _happy_ users to the App Store and _unhappy_ users to email. That selective filtering is exactly what Guideline 1.1.7 prohibits. The native `requestReview()` is only reached on one branch. **Recommend:** ask everyone via `SKStoreReviewController` (`requestReview`) and offer "Send feedback" as a peer option, not a fork that hides the rating from dissatisfied users. _Medium-high._
3. **No subscription disclosure block in the description.** The binary covers the hard 3.1.2 requirement, but reviewers expect the standard block in metadata too. Add it (copy in §4). _Medium._
4. **Empty App Review notes** (`review_information/notes.txt`). With IAP + no login, a reviewer can get confused. Add: _"No account required. Free tier = Cocktail Party briefing. Pro unlocks via standard StoreKit IAP — please test with a sandbox account. All content is app-generated daily."_ _Medium._
5. **Latent (verified safe today, worth a guard):** the `fallbackPaywall` grants Pro free via `setProForDebug(true)` whenever RevenueCat isn't configured, and a missing config makes the app run on `Briefing.sample` demo content. Your current archive bakes real keys so neither triggers — but a misconfigured archive would ship a free-Pro, placeholder-content app and get rejected under 2.1. **Recommend** a `#if !DEBUG` compile guard so the demo/free-Pro paths cannot exist in a Release binary. _Low now, cheap insurance._

## 3. Conversion: trial starts + paid conversion

The funnel is competent but leaves the two highest-leverage moves on the table:

- **No paywall in onboarding.** Onboarding (value prop → persona → team) drops the user straight into the free briefing; they never see Pro framed until they happen to tap a locked persona. **Add a trial-forward soft paywall as the final onboarding screen** ("Not now" dismissible). This is the single biggest trial-start lever for a freemium app. _High._
- **The Pro teaser is weak.** `ProPreviewSheet` shows `Briefing.sample.tlDR` — _the same generic sample for every persona_, **blurred behind a lock**, under "Here's how this room sounds." The user literally sees nothing real and nothing room-specific. **Either** show a genuine, readable, persona-specific teaser (give a real taste, then upsell) **or** make the differentiation concrete. Right now it asks for a purchase to reveal content the user can't evaluate. _High._
- **Reconsider Lifetime at launch.** Offering lifetime on a daily-habit app caps LTV before you know your pricing. Consider monthly + annual only for v1.0. _Medium._
- Working well, keep: per-context paywall headlines, 3 clean benefits, plan cards with SAVE %/BEST VALUE, trial badge gated on real eligibility, annual pre-selected, `refreshLimit` upsell card, locked Settings rows. The free/Pro gap (1 room + 1×/day vs 5 rooms + 3×/day + Local Team) is clear and defensible.

## 5. Accessibility & polish (do before a paid v1.0)

- **Dynamic Type is broken on your most important text.** `SidelineTheme.display(_)` and several `.font(.system(size:))` calls use **fixed point sizes** — the hero TL;DR, onboarding, and paywall headlines won't grow for users with larger text settings. Use relative styles (`.system(.largeTitle, design: .serif)`) or `@ScaledMetric`. _High._
- 9–10pt badges (`SAVE %`, tag pills) are below the legibility floor and also don't scale. _Medium._
- `inkTertiary` = 48% ink for captions/freshness footer likely fails WCAG AA contrast on the cream surface. Bump to ~60%. _Medium._
- VoiceOver labeling is genuinely good across persona rail, bullet cards, plan cards, and source links — no action.

## Appendix B — Specific code changes

**§2 #5 — Guard the free-Pro fallback (safety, no normal-path change).**
`Sideline/Views/PaywallView.swift`, the `fallbackPaywall` "Continue" button (~line 638): wrap the entitlement grant so it cannot exist in a Release binary.

```swift
Button {
    #if DEBUG
    if let local = entitlement as? LocalEntitlementStore {
        local.setProForDebug(true)
    }
    #endif
    dismiss()
} label: {
    Text("Continue").frame(maxWidth: .infinity)
}
```

(The fallback paywall only renders when `Purchases.isConfigured == false`, which the current archive never is — this just makes a misconfigured archive incapable of granting Pro for free.)

**§2 #2 — Review-gating funnel.** `Sideline/Views/ReviewPromptSheet.swift`. The enjoyment → reviewPitch → "Rate on the App Store" branch only surfaces the rating to users who self-identify as happy; "Not really" diverts to email. To satisfy Guideline 1.1.7, either (a) drop the pre-question and call `@Environment(\.requestReview)` / `SKStoreReviewController.requestReview` at a positive moment for everyone, or (b) make "Rate on the App Store" and "Send feedback" peer options shown to all users on the same screen. Decision required (see below).

**§3 — Conversion changes (design decisions):**
- Onboarding paywall: add a 4th, dismissible page to `Sideline/Views/OnboardingView.swift` (or present `PaywallView` on `OnboardingView` `onDisappear`) so Pro/trial is framed once before the free experience.
- Pro teaser: `Sideline/Views/ProPreviewSheet.swift` currently blurs `Briefing.sample.tlDR` (identical for every persona). Replace with a real, readable, persona-specific teaser line, then upsell.

**§5 — Dynamic Type:** `Sideline/Utilities/Theme.swift` `display(_:)` and `.font(.system(size:))` call sites use fixed point sizes. Move headlines to relative text styles (e.g. `.system(.largeTitle, design: .serif)`) or `@ScaledMetric` so they scale. Affects hero/onboarding/paywall headline layout, so verify visually.
