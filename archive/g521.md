# The Sideline — User Experience Audit & Proposed Improvements

This document analyzes the current state of **The Sideline** iOS application codebase and backend content pipeline from a user experience (UX) perspective. It highlights key disconnects between how a user operates and experiences the app versus how the underlying code behaves, and proposes high-impact, non-breaking design and technical improvements.

## 🔍 Detailed UX Audits & Recommendations

### 1. Broken "Local Team" Persona (P0)

*   **How the User Operates & Experiences This:**
    A user upgrades to Pro, navigates to settings, and selects their favorite team (e.g., "Seattle Seahawks"). They return to the home screen and tap the "Local Team" map-pin persona. They expect a daily briefing customized to the Seahawks.
    Instead, they are shown a briefing identical to the national "Cocktail Party" one, or the feed fails to load, because the app loads a generic, non-customized local feed.
*   **The Code Problem:**
    *   In [TodayBriefingViewModel.swift](file:///Users/jackwallner/sports/Shared/Services/TodayBriefingViewModel.swift#L50), `load()` hardcodes the request scope:
        ```swift
        let briefing = try await service.latestBriefing(persona: selectedPersona, scope: .national)
        ```
        It never passes `.local` or queries based on the user's selected team.
    *   In [BriefingService.swift](file:///Users/jackwallner/sports/Shared/Services/BriefingService.swift#L35), `latestBriefing` queries Supabase filtering by `scope` (eq.national or eq.local) but has no parameters to query by a specific team.
    *   The database `briefings` schema defined in [20260516214000_initial_sideline_schema.sql](file:///Users/jackwallner/sports/supabase/migrations/20260516214000_initial_sideline_schema.sql#L103) has no team name columns.
    *   The backend edge function [index.ts](file:///Users/jackwallner/sports/SupabaseFunctions/generate-briefings/index.ts#L29) loads a static list of 40 global articles and runs Gemini to generate a single generic `local_team` briefing. It has no way to gather team-specific RSS feeds or target a specific city.
*   **Suggested Improvements:**
    1.  **Client-Side Team Propagation:** Modify [TodayBriefingViewModel.swift](file:///Users/jackwallner/sports/Shared/Services/TodayBriefingViewModel.swift) to read the `@AppStorage("favoriteTeam")` value and query with `scope: selectedPersona == .localTeam ? .local : .national`.
    2.  **API / Database Update:** Modify `BriefingService` to pass the selected team name as a query parameter (e.g., `&team=eq.Seattle%20Seahawks`). Update the database `briefings` table to contain a nullable `team_name` text column.
    3.  **On-Demand Generation:** Update the backend edge function to check if a specific team briefing exists. If not, trigger an on-the-fly Gemini generation targeting that team's keywords using a Search API or specialized team RSS feeds (rather than just the generic national ESPN/AP feeds).

### 2. Locked Persona Paywall Takeover (P1)

*   **How the User Operates & Experiences This:**
    A free user is curious about what "Sports Talk for Moms" or "Date Night" briefings look like. They tap the locked chip in the top rail. Immediately, a full-screen modal sheets up blocking their screen and demanding payment. This reads as an aggressive barrier and interrupts their flow.
*   **The Code Problem:**
    *   In [TodayBriefingView.swift](file:///Users/jackwallner/sports/Sideline/Views/TodayBriefingView.swift#L20-L27), when a locked persona chip is tapped, `showingPaywall` is set to `true`:
        ```swift
        PersonaRail(selected: viewModel.selectedPersona, isPro: entitlement.isPro) { persona in
            Task {
                let didSelect = await viewModel.select(persona)
                if !didSelect {
                    showingPaywall = true
                }
            }
        }
        ```
*   **Suggested Improvements:**
    *   **Implement a Feature Preview (Teaser):** Instead of showing the full paywall modal immediately, implement the pattern suggested in [08_pro_gate.md](file:///Users/jackwallner/sports/docs/claude-design/screens-to-design/08_pro_gate.md).
    *   Let the user select the locked persona, but render the briefing feed in a **blurred/redacted state**. Overlay a clean, embedded card on top of the blur that reads:
        > 💡 **Office Watercooler**
        > *"Get safe, current takes for coworkers."*
        > [ Unlock with Pro ]
    *   This lets the user visually understand the feature's value and experience the navigation feel before they are asked to commit financially.

### 3. Hidden Tag Reasons / Explanations (P1)

*   **How the User Operates & Experiences This:**
    A user sees a talking point tagged as "Drama" or "Nice Guy" (e.g., about a player donating a check or a locker-room controversy). They want to know *why* that tag was applied. Right now, there is no way for a sighted user to read the reason. Only VoiceOver users can access the reason text.
*   **The Code Problem:**
    *   In [BulletCard.swift](file:///Users/jackwallner/sports/Sideline/Views/Components/BulletCard.swift#L81), the `tagReason` (which Gemini successfully generates on the backend) is only bound to the `.accessibilityLabel` of the tag pill:
        ```swift
        .accessibilityLabel(tagAccessibility(tag: tag, reason: bullet.tagReason))
        ```
*   **Suggested Improvements:**
    1.  **Visual Display in Bullet Cards:** Render the `tagReason` directly under the tag pill or inline inside the card (e.g., in a small, italicized gray font).
    2.  **Interactive Tooltip:** Make the tag pill tappable/hoverable, showing a popover or a small expandable drawer that reveals the reason (e.g. *"Drama: Locker-room sources are 'frustrated'"*).
    3.  **Detailed Briefing Sheet:** Implement the [BriefingDetailView.swift](file:///Users/jackwallner/sports/docs/claude-design/screens-to-design/07_briefing_detail.md) sheet (detent `.medium`) which displays the bullet details and the tag reason in full.

### 4. Inability to Copy or Share Talking Points (P1)

*   **How the User Operates & Experiences This:**
    A user finds a perfect line to drop at a party or a text to send to their kid. They want to copy the suggested question or a bullet point to their clipboard or share it via iMessage/WhatsApp. Currently, there are no share or copy actions, and they cannot highlight the text because standard SwiftUI `Text` prevents user selection.
*   **The Code Problem:**
    *   There are no context menus, copy gestures, or system share sheet triggers (`ShareLink`) anywhere in [BulletCard.swift](file:///Users/jackwallner/sports/Sideline/Views/Components/BulletCard.swift) or [SuggestedQuestionCard.swift](file:///Users/jackwallner/sports/Sideline/Views/Components/SuggestedQuestionCard.swift).
*   **Suggested Improvements:**
    1.  **Text Selection:** Add `.textSelection(.enabled)` to the talking point, tie-in, and suggested question text blocks so standard iOS copy gestures work.
    2.  **Action Buttons:** Add a small, low-opacity copy or share icon next to the source link on cards. Tapping it triggers a clipboard copy (with a brief visual checkmark state) or opens a system share sheet using SwiftUI `ShareLink`:
        ```swift
        ShareLink(item: bullet.talkingPoint, preview: SharePreview("Talking Point", image: Image("AppIcon"))) {
            Label("Share", systemImage: "square.and.arrow.up")
        }
        ```

### 5. Persona Switching causes Full-Screen Flash (P1)

*   **How the User Operates & Experiences This:**
    When the user switches between different personas in the top rail, the app completely wipes the existing content and flashes a skeleton loading state. This disrupts visual continuity and makes the app feel sluggish, even when switching to cached content.
*   **The Code Problem:**
    *   In [TodayBriefingViewModel.swift](file:///Users/jackwallner/sports/Shared/Services/TodayBriefingViewModel.swift#L47), calling `load()` instantly sets `state = .loading`, clearing the current briefing:
        ```swift
        public func load() async {
            state = .loading
            ...
        }
        ```
*   **Suggested Improvements:**
    1.  **Background Loading with Overlay:** Modify the view model to keep the `lastBriefing` visible in the `.populated` state while loading the new briefing in the background. Show a subtle progress indicator (like a thin loading bar at the top of the feed) instead of showing the full-screen skeleton.
    2.  **Smooth Transitions:** When the new briefing loads, use a cross-fade transition (`.transition(.opacity)`) with an ease-out animation to swap the contents smoothly.

### 6. Jarring Refresh Limit Layout Shift (P2)

*   **How the User Operates & Experiences This:**
    A free user pulls the feed down to refresh. Because they are on the free tier and already have the daily update, the app injects a large "You're caught up" card at the very top of the scroll view. This pushes the briefing content down instantly, which feels like a buggy visual jump.
*   **The Code Problem:**
    *   In [TodayBriefingView.swift](file:///Users/jackwallner/sports/Sideline/Views/TodayBriefingView.swift#L75), when state is `.refreshLimit`, the `refreshLimitCard` is injected above the briefing card layout:
        ```swift
        case .refreshLimit:
            refreshLimitCard
            if let briefing = viewModel.lastBriefing {
                briefingView(briefing, isOffline: false)
            }
        ```
*   **Suggested Improvements:**
    1.  **Non-Intrusive HUD/Toast:** Instead of injecting a card in the scroll view, show a floating HUD toast at the top or bottom of the screen (e.g. "Caught up! Pro refreshes 3× daily.") that automatically fades out after 3 seconds.
    2.  **Footer Integration:** Add the refresh status inside the [FreshnessFooter.swift](file:///Users/jackwallner/sports/Sideline/Views/Components/FreshnessFooter.swift) or inside the Navigation Bar title subtitle rather than as a primary body block.

### 7. Personalization Restricted to NFL Only (P2)

*   **How the User Operates & Experiences This:**
    A user wants to talk to a fan who follows basketball, baseball, or hockey. They navigate to Settings -> "Local team" to change their preference, only to find a list consisting entirely of the 32 NFL teams. Users in NBA-focused cities or non-football fans feel excluded.
*   **The Code Problem:**
    *   In [SettingsView.swift](file:///Users/jackwallner/sports/Sideline/Views/SettingsView.swift#L92), the `teams` array is hardcoded to a static list of 32 NFL teams:
        ```swift
        private let teams = [
            "Arizona Cardinals", "Atlanta Falcons", ..., "Washington Commanders"
        ]
        ```
*   **Suggested Improvements:**
    1.  **Multi-Sport Support:** Expand the teams array to include major teams from the NBA, MLB, NHL, MLS, and top European soccer clubs.
    2.  **Categorized List:** Organize the list into sections by sport (e.g., "Football (NFL)", "Basketball (NBA)") to maintain order.
    3.  **Search Filtering:** Add a simple `.searchable` text field to the `TeamPickerView` list so users can type to quickly find their team instead of scrolling through 100+ items.

### 8. Offline Blank States on First Launch (P2)

*   **How the User Operates & Experiences This:**
    A user downloads the app and opens it on an airplane or subway before their first network request completes. They are shown a blank screen with a "Nothing fresh yet" warning and a "Try again" button that does nothing offline.
*   **The Code Problem:**
    *   If no network connection exists and the SwiftData cache is empty, [TodayBriefingView.swift](file:///Users/jackwallner/sports/Sideline/Views/TodayBriefingView.swift#L163) falls back to `emptyState`.
*   **Suggested Improvements:**
    *   **Fallback to Local Sample Briefing:** The codebase already includes [SampleBriefingService.swift](file:///Users/jackwallner/sports/Shared/Services/BriefingService.swift#L78) which yields `Briefing.sample`.
    *   If loading fails and the local database cache is empty, the app should load `Briefing.sample` and display it with a friendly overlay:
        > 🛜 **Welcome to The Sideline (Demo Mode)**
        > *You are offline. Here is an example of what your daily sports talking points will look like once you connect.*
    *   This provides immediate value and demonstrates the app's core utility rather than starting with a broken empty screen.

## 🛠️ Micro-Aesthetic & Polish Improvements

1.  **Amber Text Contrast:**
    *   The color token `SidelineTheme.amberText` (`#523600` on line 6 of [Theme.swift](file:///Users/jackwallner/sports/Sideline/Utilities/Theme.swift)) is used on the suggested question card's eyebrow text ("Your move"). It is highly readable in light mode, but has poor contrast in dark mode against the dark card backgrounds.
    *   *Fix:* Define light and dark variants for `amberText` in an asset catalog or dynamically swap color values based on color scheme.
2.  **Source Link Truncation:**
    *   In [BulletCard.swift](file:///Users/jackwallner/sports/Sideline/Views/Components/BulletCard.swift#L91), the source link text can truncate aggressively:
        ```swift
        Text(bullet.sourceHeadline)
            .lineLimit(1)
            .truncationMode(.tail)
        ```
    *   If the source headline is long, it becomes completely illegible (e.g. *"Veteran QB benched amid playoff push..."*).
    *   *Fix:* Increase the line limit to 2 for source links, or use a smaller font style to allow the headline to remain descriptive.
